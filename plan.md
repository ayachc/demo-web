# 资产借用和归还 Web 服务设计方案

## 1. 目标

构建一个包含前端、后端和日志模块的资产借用/归还 Web 服务。主页展示所有资产的当前状态，用户可以通过弹窗完成添加资产、借用资产、归还资产，提交后页面立即刷新数据。

## 2. 技术选型

- 后端：FastAPI
- 数据校验：Pydantic
- 前端：原生 HTML、CSS、JavaScript
- 数据存储：进程内内存存储，预置 4 个资产和借用状态
- 静态资源托管：FastAPI StaticFiles
- 日志：Python logging，按用途输出到 `logs/app.log`、`logs/access.log`、`logs/error.log`

选择原因：

- 项目规模较小，FastAPI 可以快速提供清晰的 API、自动数据校验和可维护的模块结构。
- 原生前端足以覆盖当前交互，避免引入构建工具和额外依赖。
- 当前需求未要求持久化数据库，内存存储能降低复杂度；后续可以在服务层下方替换为 SQLite、PostgreSQL 或其他数据库。

## 3. 功能范围

### 3.1 主页资产展示

主页展示所有资产，字段包括：

- 资产编号
- 资产名称
- 资产图片
- 总量
- 剩余数量
- 借用人工号列表

初始预置 4 个资产：

- LAPTOP-001：研发笔记本
- MONITOR-001：显示器
- CAMERA-001：会议摄像头
- ROUTER-001：测试路由器

每个资产带有预置总量、剩余数量、图片和部分借用人工号。

### 3.2 添加资产

点击“添加资产”按钮后打开弹窗，填写：

- 资产编号
- 资产名称
- 图片地址
- 总量

提交规则：

- 资产编号不能为空且不能重复。
- 总量必须是正整数。
- 新资产初始剩余数量等于总量。
- 借用人工号列表为空。

### 3.3 借用资产

点击“借用”按钮后打开弹窗，填写：

- 资产编号
- 借用人工号

提交规则：

- 资产必须存在。
- 剩余数量必须大于 0。
- 同一工号不能重复借用同一个资产。
- 借用成功后剩余数量减 1，借用人工号加入列表。

### 3.4 归还资产

点击“归还”按钮后打开弹窗，填写：

- 资产编号
- 借用人工号

提交规则：

- 资产必须存在。
- 借用人工号必须在该资产借用列表中。
- 归还成功后剩余数量加 1，借用人工号从列表移除。

## 4. API 设计

### 4.1 获取资产列表

- 方法：`GET`
- 路径：`/api/assets`
- 响应：资产数组

### 4.2 添加资产

- 方法：`POST`
- 路径：`/api/assets`
- 请求体：

```json
{
  "asset_id": "KEYBOARD-001",
  "name": "机械键盘",
  "image_url": "https://example.com/image.jpg",
  "total": 10
}
```

### 4.3 借用资产

- 方法：`POST`
- 路径：`/api/assets/{asset_id}/borrow`
- 请求体：

```json
{
  "employee_id": "E1001"
}
```

### 4.4 归还资产

- 方法：`POST`
- 路径：`/api/assets/{asset_id}/return`
- 请求体：

```json
{
  "employee_id": "E1001"
}
```

### 4.5 健康检查

- 方法：`GET`
- 路径：`/health`
- 响应：服务状态和当前资产数量

## 5. 后端模块设计

```text
app/
  __init__.py
  main.py              # FastAPI 应用入口、路由注册、静态页面托管
  models.py            # Pydantic 请求/响应模型
  store.py             # 内存数据与业务操作
  logging_config.py    # 日志格式、handler、request_id 上下文
static/
  index.html
  styles.css
  app.js
logs/
  .gitkeep
```

业务状态由 `AssetStore` 统一管理。所有增删改操作经过服务层，避免路由中散落状态修改逻辑。

## 6. 日志设计

日志目标：

- 能定位每一次请求。
- 能看到每次业务操作的输入、结果和失败原因。
- 异常保留堆栈，便于复现和修复。

### 6.1 request_id

每个请求进入后生成或复用 `X-Request-ID`，并写入响应头。日志格式中包含 `request_id`。

### 6.2 日志文件

- `logs/app.log`：应用启动、业务操作、普通运行日志。
- `logs/access.log`：请求方法、路径、状态码、耗时、客户端 IP。
- `logs/error.log`：未处理异常和错误堆栈。

### 6.3 关键业务日志

添加资产：

- 成功：记录资产编号、名称、总量。
- 失败：记录重复编号或参数错误。

借用资产：

- 成功：记录资产编号、工号、剩余数量。
- 失败：记录资产不存在、库存不足、重复借用。

归还资产：

- 成功：记录资产编号、工号、剩余数量。
- 失败：记录资产不存在、工号未借用该资产。

## 7. 前端设计

页面结构：

- 顶部工具栏：标题、刷新、添加资产、借用、归还按钮。
- 资产列表：响应式卡片网格。
- 卡片内容：图片、资产编号、名称、库存进度、总量/剩余、借用人工号标签。
- 弹窗表单：根据动作显示不同字段。
- Toast 提示：显示成功或失败信息。

交互流程：

1. 页面加载时调用 `GET /api/assets`。
2. 用户点击按钮打开对应弹窗。
3. 表单提交后调用对应 API。
4. 成功后关闭弹窗并刷新资产列表。
5. 失败时显示错误信息，保留弹窗内容方便修正。

## 8. 错误处理

- 后端用 HTTP 状态码表达错误类型：
  - `400`：参数不合法或业务规则不满足。
  - `404`：资产不存在。
  - `409`：资产编号冲突或重复借用。
  - `500`：服务端未处理异常。
- 前端读取响应中的 `detail` 字段并展示。
- 未处理异常由统一 middleware 记录到 `error.log`，并返回通用错误响应。

## 9. 验证计划

- 启动服务：`python -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000`
- 检查健康接口：`GET /health`
- 检查资产列表：`GET /api/assets`
- 验证添加资产、借用、归还 API。
- 打开主页确认数据展示和弹窗交互可用。
